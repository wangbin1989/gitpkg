# GitPkg 安装脚本设计文档

**版本**: 1.0  
**日期**: 2026-06-21  
**状态**: 正式  

---

## 1. 概述

### 1.1 编写目的

本文档描述 GitPkg 安装脚本（`install.sh` / `install.ps1`）的架构设计、执行流程与关键决策，作为脚本维护与后续迭代的参考。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **一键安装** | 用户通过一行命令完成下载、安装、PATH 配置全流程 |
| **跨平台** | 支持 macOS (arm64)、Linux (x86_64/arm64)、Windows (x86_64/arm64) |
| **零依赖** | 仅依赖系统预装工具（curl/tar 或 PowerShell），无需额外安装 |
| **安全性** | 使用 HTTPS 下载、临时目录隔离、路径转义防注入 |
| **幂等性** | 重复执行不会产生副作用，已有安装会被覆盖并提示 |

### 1.3 脚本文件

| 文件 | 目标平台 | Shell | 调用方式 |
|------|----------|-------|----------|
| `install.sh` | macOS / Linux | Bash (POSIX) | `curl -fsSL ... \| bash` |
| `install.ps1` | Windows | PowerShell 5.1+ | `Invoke-Expression (Invoke-RestMethod ...)` |

## 2. 架构设计

### 2.1 整体流程

```
用户执行脚本
     │
     ▼
┌─────────────┐
│  平台检测    │  uname -s / uname -m / PROCESSOR_ARCHITECTURE
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  依赖检查    │  curl, tar (Unix) / PowerShell 内置 cmdlet (Windows)
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│  获取版本信息    │  GitHub API: /releases/latest 或 /releases/tags/{version}
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  匹配平台资产    │  按 OS + Arch 筛选 release asset
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  下载到临时目录  │  curl / Invoke-WebRequest
└──────┬──────────┘
       │
       ▼
┌─────────────────┐
│  解压并安装      │  tar -xzf / Expand-Archive → mv/Copy-Item → ~/.gitpkg/bin/
└──────┬──────────┘
       │
       ▼
┌─────────────────────┐
│  PATH 提示          │  输出 eval "$(gitpkg init <shell>)" 供用户执行
│                     │  提示可追加到 shell 配置文件永久生效
└──────┬──────────────┘
       │
       ▼
    安装完成
```

### 2.2 目录结构

安装完成后，GitPkg 的文件布局：

```
~/.gitpkg/
├── bin/               # gitpkg 可执行文件所在目录
│   └── gitpkg         # (或 gitpkg.exe)
├── tools/             # (后续由 gitpkg install 管理)
├── manifest.json      # (后续由 gitpkg install 管理)
└── tmp/               # (后续由 gitpkg install 管理)
```

安装脚本仅负责将 `gitpkg` 二进制放入 `~/.gitpkg/bin/`，不创建其他目录。`tools/`、`manifest.json`、`tmp/` 由 `gitpkg install` 命令在首次使用时自动创建。

## 3. install.sh 详细设计

### 3.1 配置常量

```bash
REPO="wangbin1989/gitpkg"
GITHUB_API="https://api.github.com/repos/${REPO}"
INSTALL_DIR="${HOME}/.gitpkg/bin"
BINARY_NAME="gitpkg"
```

| 常量 | 说明 |
|------|------|
| `REPO` | GitHub 仓库 `owner/repo`，用于拼接 API 地址 |
| `GITHUB_API` | GitHub API 基础 URL |
| `INSTALL_DIR` | 安装目标目录，固定为 `~/.gitpkg/bin` |
| `BINARY_NAME` | 可执行文件名，固定为 `gitpkg` |

### 3.2 平台检测 (`detect_platform`)

```
输入: 无
输出: "{os}-{arch}" 字符串
支持组合: darwin-arm64, linux-x86_64, linux-arm64
```

**OS 映射**:

| `uname -s` | 输出 |
|------------|------|
| `Darwin` | `darwin` |
| `Linux` | `linux` |
| 其他 | 报错退出 |

**Arch 映射**:

| `uname -m` | 输出 |
|------------|------|
| `x86_64` / `amd64` | `x86_64` |
| `arm64` / `aarch64` | `arm64` |
| 其他 | 报错退出 |

**设计决策**: 不支持 Windows（由 `install.ps1` 覆盖）和 Intel macOS（已从 CI 矩阵移除）。

### 3.3 下载与安装 (`download_and_install`)

```
输入: platform (string), version (string)
输出: 二进制安装到 INSTALL_DIR
```

**执行步骤**:

1. **版本信息获取**
   - `version == "latest"` → 调用 `/releases/latest`
   - 否则 → 调用 `/releases/tags/{version}`
   - 从 JSON 响应中用 `grep` + `cut` 提取 `browser_download_url`
   - 筛选条件: 文件名匹配 `gitpkg-.*-{platform}.tar.gz`

2. **下载**
   - 创建临时目录 `mktemp -d`
   - 注册 `trap cleanup EXIT` 确保脚本退出时清理临时目录
   - 使用 `curl -fSL --progress-bar` 下载（`-f` 失败时返回非零退出码，`-S` 显示错误，`-L` 跟随重定向）

3. **解压**
   - `tar -xzf` 解压到临时目录
   - 校验 `gitpkg` 二进制文件存在

4. **安装**
   - `mkdir -p` 创建安装目录
   - `chmod +x` 设置可执行权限
   - `mv` 移动到目标路径（已有文件会被覆盖，给出警告）

5. **验证**
   - 执行 `gitpkg --version` 确认安装成功

### 3.4 临时目录与清理

```bash
tmp_dir=$(mktemp -d)
cleanup() { rm -rf "${tmp_dir:-}"; }
trap cleanup EXIT
```

**设计决策**: 使用函数封装 `cleanup()` 而非内联命令，原因：
- 临时目录路径可能包含单引号等特殊字符，内联 `trap 'rm -rf ...' EXIT` 会导致引号错乱
- 使用 `${tmp_dir:-}` 默认值语法，防止 `set -u` 模式下变量未绑定报错

### 3.5 PATH 提示 (`check_path`)

```
输入: 无
输出: 输出初始化命令供用户手动执行
```

**执行步骤**:

1. **检测当前 Shell**
   - 读取 `$SHELL` 环境变量获取默认 Shell
   - 映射到对应的配置文件路径

2. **检查是否已在 PATH 中**
   - `":$PATH:"` 模式匹配，避免前缀/后缀误匹配
   - 若已在 PATH 中，提示用户重新打开终端或运行 `source <rc_file>` 使新安装的二进制生效
   - 若不在 PATH 中，继续以下步骤

3. **输出初始化命令** — 根据 Shell 类型输出不同语法:

   | Shell | 初始化命令 | 自动补全命令 |
   |-------|-----------|-------------|
   | zsh | `eval "$(gitpkg init zsh)"` | `eval "$(gitpkg completion zsh)"` |
   | bash | `eval "$(gitpkg init bash)"` | `eval "$(gitpkg completion bash)"` |
   | fish | `gitpkg init fish \| source` | `gitpkg completion fish > ~/.config/fish/completions/gitpkg.fish` |
   | PowerShell | `Invoke-Expression (& 'gitpkg' init powershell)` | `Invoke-Expression (& 'gitpkg' completion powershell)` |

   - 提示用户可将命令追加到 shell 配置文件中使其永久生效
   - 除 PATH 初始化外，单独输出自动补全命令供用户按需启用

**设计决策**: 不自动修改用户的 shell 配置文件，仅输出初始化和自动补全命令。理由：
- 避免未经用户明确授权修改其配置文件
- `gitpkg init <shell>` 配置 PATH 和 GITPKG_HOME，`gitpkg completion <shell>` 配置自动补全，职责分离
- 用户可选择仅在当前会话生效或写入配置文件永久生效
- fish 不支持 `eval "$(…)"` 语法，需使用 `… | source` 管道形式

### 3.6 主流程 (`main`)

```
main(version):
    打印 Banner
    输出仓库信息和安装目录
    platform = detect_platform()
    检查 curl、tar 是否可用
    download_and_install(platform, version)
    check_path()
    打印安装成功提示
```

**入口**: `main "$@"` — 将命令行参数传递给 `main`，支持 `bash -s v1.1.4` 指定版本。

## 4. install.ps1 详细设计

### 4.1 与 install.sh 的对应关系

| install.sh | install.ps1 | 说明 |
|------------|-------------|------|
| `detect_platform()` | `Get-Platform` | 使用 `$env:PROCESSOR_ARCHITECTURE` |
| `curl` | `Invoke-RestMethod` / `Invoke-WebRequest` | PowerShell 内置 |
| `tar -xzf` | `Expand-Archive` | 仅支持 zip 格式 |
| `mktemp -d` | `Join-Path $env:TEMP "gitpkg-install-$(Get-Random)"` | 随机临时目录 |
| `trap cleanup EXIT` | `Remove-Item -Recurse -Force $tmpDir` | 显式清理（无 trap 等价物） |
| 输出初始化命令 | 输出初始化命令 | 两者行为一致（fish 用 `\| source`） |

### 4.2 平台检测

```powershell
$arch = switch ($env:PROCESSOR_ARCHITECTURE) {
    "AMD64" { "x86_64" }
    "ARM64" { "arm64" }
}
return "windows-$arch"
```

仅支持 Windows 平台，不检测 OS（PowerShell 脚本本身即限定 Windows）。

### 4.3 PATH 提示策略

与 install.sh 一致，PowerShell 版本仅输出 `eval` 命令，不自动修改 `$PROFILE`。

1. 检查当前用户 PATH 是否已包含 `$InstallDir`
2. 若已在 PATH 中，提示用户重新打开终端或运行 `. $PROFILE` 使新安装的二进制生效
3. 若不在 PATH 中，输出 `Invoke-Expression (& 'gitpkg' init powershell)` 供用户手动执行

### 4.4 归档格式差异

| 平台 | 归档格式 | 解压工具 |
|------|----------|----------|
| macOS / Linux | `.tar.gz` | `tar -xzf` |
| Windows | `.zip` | `Expand-Archive` |

## 5. GitHub API 交互

### 5.1 API 端点

| 场景 | 端点 |
|------|------|
| 获取最新版本 | `GET /repos/{owner}/{repo}/releases/latest` |
| 获取指定版本 | `GET /repos/{owner}/{repo}/releases/tags/{tag}` |

### 5.2 资产筛选

从 Release 的 `assets` 数组中筛选匹配当前平台的文件：

**命名模式**: `gitpkg-{version}-{platform}.{ext}`

| 平台 | 匹配后缀 | 扩展名 |
|------|----------|--------|
| macOS arm64 | `darwin-arm64` | `.tar.gz` |
| Linux x86_64 | `linux-x86_64` | `.tar.gz` |
| Linux arm64 | `linux-arm64` | `.tar.gz` |
| Windows x86_64 | `windows-x86_64` | `.zip` |
| Windows arm64 | `windows-arm64` | `.zip` |

### 5.3 响应解析

**install.sh**: 使用 `grep` + `cut` 从 JSON 中提取 `browser_download_url`：
```bash
curl -fsSL "${api_url}" \
    | grep "browser_download_url" \
    | grep "gitpkg-.*-${archive_suffix}\.${archive_ext}" \
    | head -n 1 \
    | cut -d '"' -f 4
```

**install.ps1**: 使用 `Invoke-RestMethod` 自动解析 JSON，通过 `Where-Object` 筛选：
```powershell
$releaseJson = Invoke-RestMethod -Uri $apiUrl
$asset = $releaseJson.assets | Where-Object {
    $_.name -match "gitpkg-.*-$platformSuffix\.zip"
}
```

## 6. 安全设计

### 6.1 传输安全

- 所有网络请求强制 HTTPS
- `curl -f` 在 HTTP 错误时返回非零退出码，配合 `set -e` 终止脚本

### 6.2 临时文件安全

- 使用 `mktemp -d` 创建不可预测的临时目录名
- `trap EXIT` 确保任何退出路径都会清理临时文件
- 安装完成前，二进制文件始终在临时目录中，不会残留

### 6.3 路径注入防护

**威胁**: 用户 HOME 路径包含单引号等特殊字符时，eval 执行初始化脚本可能导致命令注入。

**防护措施**:
1. 路径用单引号包裹（防止 `$`、`` ` ``、`"`、`\` 展开）
2. 路径中的单引号通过 `'\''` 模式转义（标准 POSIX 转义方式）
3. PowerShell 使用 `''` 双写转义单引号

### 6.4 错误处理

- `set -euo pipefail` (install.sh): 任何命令失败立即退出，未绑定变量报错，管道中任意命令失败视为整体失败
- `$ErrorActionPreference = "Stop"` (install.ps1): 任何 cmdlet 抛出异常时终止执行

## 7. CI/CD 集成

### 7.1 构建矩阵

安装脚本与 GitHub Actions release 工作流的构建矩阵对齐：

| 平台标识 | CI Runner | RID | 归档格式 |
|----------|-----------|-----|----------|
| `linux-x86_64` | `ubuntu-latest` | `linux-x64` | `.tar.gz` |
| `linux-arm64` | `ubuntu-24.04-arm` | `linux-arm64` | `.tar.gz` |
| `darwin-arm64` | `macos-latest` | `osx-arm64` | `.tar.gz` |
| `windows-x86_64` | `windows-latest` | `win-x64` | `.zip` |
| `windows-arm64` | `windows-11-arm` | `win-arm64` | `.zip` |

### 7.2 发布产物命名

```
gitpkg-v{version}-{platform}.{tar.gz|zip}
```

示例: `gitpkg-v2.0.0-darwin-arm64.tar.gz`

### 7.3 校验和文件

CI 自动生成 `checksums.txt`，包含所有归档文件的 SHA256 值，随 Release 一起发布。安装脚本当前不校验此文件（校验由 `gitpkg install` 命令在安装工具时执行）。

## 8. 设计决策记录

### D-01: 为什么使用 `grep` 而非 `jq` 解析 JSON？

**决策**: install.sh 使用 `grep` + `cut` 解析 GitHub API 响应。

**理由**:
- `jq` 不是所有系统的预装工具，额外安装依赖违反"零依赖"目标
- GitHub Release API 的 JSON 结构稳定，`browser_download_url` 字段的提取逻辑足够可靠
- `grep` + `cut` 在所有 POSIX 系统上可用

**权衡**: 缺乏严格的 JSON 解析能力，若 GitHub API 响应格式变更可能导致解析失败。已通过错误检查（URL 为空时报错）缓解。

### D-02: 为什么只输出初始化命令而不自动修改配置文件？

**决策**: 安装完成后仅输出初始化命令（zsh/bash 用 `eval "$(...)"`, fish 用 `... | source`），不自动修改 shell 配置文件。

**理由**:
- 避免未经用户明确授权修改其 `.zshrc` / `.bashrc` 等配置文件
- `gitpkg init <shell>` 已将 PATH 配置和自动补全合并为一条命令，用户只需复制一次
- 用户可选择仅在当前会话生效（直接执行）或永久生效（追加到配置文件）
- 简化安装脚本逻辑，消除路径转义、文件写入等复杂处理

### D-03: 为什么使用函数封装 trap？

**决策**: `cleanup() { rm -rf "${tmp_dir:-}"; }` 而非 `trap 'rm -rf "${tmp_dir}"' EXIT`。

**理由**:
- 临时目录路径由 `mktemp` 生成，可能包含单引号
- 内联 trap 中的单引号会与路径中的单引号冲突，导致 shell 语法错误
- 函数封装避免了引号嵌套问题，且 `${tmp_dir:-}` 防止 `set -u` 报错

### D-04: 为什么安装到 `~/.gitpkg/bin/` 而非 `/usr/local/bin/`？

**决策**: 安装到用户目录而非系统目录。

**理由**:
- 不需要 `sudo` 权限，降低使用门槛
- 与 GitPkg 自身的工具管理目录结构一致（`~/.gitpkg/bin/` 用于存放所有已安装工具的链接）
- 避免污染系统目录，卸载时无需提权

### D-05: 为什么同时维护 install.sh 和 install.ps1？

**决策**: 为 Unix 和 Windows 分别维护独立的安装脚本。

**理由**:
- Bash 和 PowerShell 的语法、命令、错误处理机制完全不同
- 共享逻辑会导致大量平台判断分支，可读性差
- 两个脚本各自约 200 行，维护成本可接受
- 调用方式不同（管道执行 vs 直接执行），统一困难

## 9. 已知限制

| 编号 | 限制 | 影响 | 缓解措施 |
|------|------|------|----------|
| L-01 | 不支持 Intel macOS | 少数 Intel Mac 用户无法使用 | CI 已移除 Intel 构建；用户可手动安装 |
| L-02 | 无 GitHub API Token 支持 | 高频使用可能触发速率限制 (60次/小时) | 单次安装仅 1-2 次 API 调用，影响有限 |
| L-03 | 安装脚本不校验 checksums.txt | 下载的二进制未经完整性验证 | GitHub HTTPS 传输提供基础保护；后续可增加 |
| L-04 | 不自动配置 PATH | 用户需手动执行 eval 命令 | 输出清晰的 eval 命令供复制，降低操作门槛 |
| L-05 | 不支持代理配置 | 企业网络环境可能无法访问 GitHub | 后续可增加 `HTTPS_PROXY` 环境变量检测 |

## 10. 后续演进方向

1. **checksums.txt 校验**: 安装脚本下载二进制后自动校验 SHA256，与 `gitpkg install` 行为一致
2. **代理支持**: 检测 `HTTPS_PROXY` 环境变量并传递给 curl
3. **GitHub Token**: 支持 `GITHUB_TOKEN` 环境变量以提升 API 速率限制
4. **自更新机制**: `gitpkg self-update` 调用安装脚本实现自身更新（已部分实现）
5. **Windows 归档格式统一**: 考虑同时发布 `.tar.gz` 格式，减少平台差异
